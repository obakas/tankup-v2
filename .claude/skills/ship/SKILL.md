# Ship

Automates the branch → PR → auto-merge flow for this repo's `main-protection` ruleset (requires a PR + `backend-test`, `frontend-test`, `mobile-lint` all passing). Follow these steps in order. Stop and report back to the user at any step that fails rather than pushing forward.

## 1. Check there's something to ship

Run `git status`. If there is nothing staged, unstaged, or untracked that looks like real work (ignore stray build artifacts, caches), tell the user there's nothing to ship and stop.

## 2. Branch handling

- If currently on `main`: create a new branch off it with a short kebab-case name summarizing the diff, matching this repo's existing convention (e.g. `fix-frontend-no-explicit-any`, `add-github-actions-ci`).
- If already on a feature branch: commit onto that branch rather than creating a nested branch.
- Never commit or push directly to `main` — the ruleset will reject it anyway, but don't attempt it.

## 3. Stage carefully

Review `git status` output before staging. Apply the same judgment as the top-level git-safety instructions:
- Don't blindly `git add -A` or `git add .`.
- Exclude unrelated pre-existing changes that aren't part of what the user is shipping.
- Exclude generated/build artifacts (e.g. `frontend/dist`, `mobile/build-*.apk`, `.pytest_cache`, `__pycache__`) unless the user specifically changed them on purpose.
- If anything staged looks like it could be a secret (`.env`, credentials, keys) even under an innocuous filename, stop and check its contents before proceeding.

## 4. Pre-flight local checks

Before pushing, run the check(s) for whichever workspace(s) actually changed (compare against `origin/main`):

| Changed path | Command |
|---|---|
| `backend/**` | `cd backend && source .venv/bin/activate && pytest` |
| `frontend/**` | `cd frontend && npm run lint && npm run test && npm run build` |
| `mobile/**` | `cd mobile && npm run lint` |

If a check fails, stop and report the failure to the user instead of pushing. Don't open a PR that's known to fail CI — that just wastes an auto-merge cycle that will never complete. If you can fix the failure yourself confidently (e.g. a lint error, an obvious type error), fix it and re-run before continuing; otherwise report it and ask.

Clean up any local artifacts these checks generate before committing (e.g. `frontend/dist` gets regenerated by `npm run build` — revert it if it's tracked and unrelated to this change, `__pycache__` / `.pytest_cache` should never be committed).

## 5. Commit

Write a commit message following this repo's established convention: concise, explains *why* not *what*, ends with:
```
Co-Authored-By: Claude Sonnet 5 <noreply@anthropic.com>
```

## 6. Push

```
git push -u origin <branch>
```
(use `git push` with no `-u` if the branch already has an upstream)

## 7. Open the PR

```
gh pr create --title "<short title>" --body "$(cat <<'EOF'
## Summary
- ...

## Test plan
- [x] <local checks that were run and passed>

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

## 8. Arm auto-merge

```
gh pr merge --auto --squash
```

This schedules the merge — GitHub completes it automatically once `backend-test`, `frontend-test`, and `mobile-lint` all report success. If a required check fails, GitHub cancels auto-merge and leaves the PR open; nothing broken reaches `main`. Squash keeps `main` history as one commit per shipped change.

If `gh pr merge --auto` errors because auto-merge isn't enabled on the repo, tell the user to enable "Allow auto-merge" in Settings → General → Pull Requests, or run:
```
gh api -X PATCH repos/obakas/tankup-v2 -f allow_auto_merge=true
```

## 9. Report back

Give the user the PR URL and confirm auto-merge is armed and waiting on checks. Don't poll or wait for the merge to complete — that happens asynchronously on GitHub's side.
